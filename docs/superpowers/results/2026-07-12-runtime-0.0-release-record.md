## Sanitized live smoke (paid, opt-in)

Model `deepseek-v4-flash`, `wire_api = "chat_completions"`, isolated
`CODEX_HOME`, run via `codex exec --json`. No secrets or private prompt/tool
content appear below.

### Text + usage

```json
{"type":"item.completed","item":{"id":"item_0","type":"agent_message","text":"ACK"}}
{"type":"turn.completed","usage":{"input_tokens":7915,"cached_input_tokens":2304,"output_tokens":73,"reasoning_output_tokens":0}}
```

Confirms: a full turn completes on the DeepSeek chat path, the assistant message
lands, and provider-reported usage — including nonzero `cached_input_tokens` —
reaches the CLI's public JSON output with no new flag.

### Prompt-cache steady state

See `2026-07-12-deepseek-cache-baseline.md`: three reproducible warm-cache
steady-state runs on the `normal` profile show ~98% harness hit rate, with
`system_changed` / `tools_changed` / `history_rewritten` all false and a
`WithinTolerance` diagnosis — no evidence to change prompt/tool/history
structure.

### Long-context checkpoints (live, token-calibrated)

All three §5.4 long-context checkpoints run against real DeepSeek via
`DEEPSEEK_CACHE_PROFILE=context{100k,300k,700k} DEEPSEEK_CACHE_ROUNDS=2 just
test-deepseek-cache`. The live benchmark calibrates the stable document against
provider-reported `prompt_tokens` (rescaling until within ±10% of target), so
these are real provider-token sizes, not char estimates:

| profile | target | measured (control) | measured (harness) | steady-state cache |
| --- | ---: | ---: | ---: | ---: |
| context100k | 100,000 | 99,964 | 101,783 | ~100% |
| context300k | 300,000 | 299,996 | 301,814 | ~100% |
| context700k | 700,000 | 699,997 | 701,816 | ~100% |

The **control** value is what the calibration loop targets, so it lands within
±0.05% of the profile target; the **harness** carries the additional Codex
system prompt + tools, so it runs ~0.3%–1.8% above target (still well under the
850K compact threshold). Each checkpoint confirms long-context request
construction, HTTP body, DeepSeek receipt, and long-context response.

The `warmup` rows differ between scenarios by design. The calibration probes
already pre-warm the **control** prefix, so the control warm-up is already a near
full cache hit. The **harness** warm-up remains cold for the harness-specific
layout (system + tools differ from the control), then steady-state reaches ~100%
with a `WithinTolerance` diagnosis. Example (700K):

```
scenario  phase         round  prompt   hit      miss   hit_rate
control   warmup        0      699997   699904   93     1.000   (pre-warmed by calibration)
control   steady_state  1..2   699997   699904   93     1.000
harness   warmup        0      701816   1664     700152 0.002   (cold: harness layout)
harness   steady_state  1..2   ~701840  ~701760  <140   1.000
```


### Tool call and continuation (live)

Live `codex exec --json`, `danger-full-access` sandbox, verified at both
`medium` (default) and `high` reasoning effort:

```json
{"type":"item.completed","item":{"type":"command_execution","command":"/bin/zsh -lc 'echo hello-high'","aggregated_output":"hello-high\n","exit_code":0,"status":"completed"}}
{"type":"item.completed","item":{"type":"agent_message","text":"The command output is `hello-high`."}}
{"type":"turn.completed","usage":{"input_tokens":15927,"cached_input_tokens":15744,"output_tokens":92,"reasoning_output_tokens":0}}
```

Confirms end-to-end on real DeepSeek: the model emits a tool call, the shell
tool executes, the result is fed back, and the continuation turn completes —
identically at `medium` (thinking enabled) and `high` (annotated 8,192 budget).
Also covered deterministically offline
(`chat_completions_tool_call_then_continuation`,
`chat_completions_persist_resume_continue`).

## Post-0.0 change policy

Per §14: accept fixes for DeepSeek, context correctness, tools, compaction,
persistence, app-server lifecycle, security, and data loss; require a Runtime
0.0 regression test for every supported-path behavior change; do not merge
upstream wholesale, add other-vendor branches, start feature-gated dependency
cutting, add enterprise multi-tenancy, or change the app-server protocol without
an application requirement.
