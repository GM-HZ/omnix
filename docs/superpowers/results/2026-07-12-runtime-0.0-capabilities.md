# DeepSeek Application Agent Runtime 0.0 — Supported Capabilities & Limitations

Companion to the design freeze
(`../specs/2026-07-12-deepseek-application-agent-runtime-0.0-design.md`). This
document records what Runtime 0.0 supports, the recommended configuration, the
release-gate evidence, and the known limitations.

## Recommended DeepSeek configuration

```toml
model = "deepseek-v4-flash"
model_provider = "deepseek"

[model_providers.deepseek]
name = "DeepSeek"
base_url = "https://api.deepseek.com/v1"
env_key = "DEEPSEEK_API_KEY"
wire_api = "chat_completions"
```

| Setting | Value |
| --- | --- |
| Release-gated model | `deepseek-v4-flash` |
| Same-protocol, not separately gated | `deepseek-v4-pro` |
| Wire API | `POST /chat/completions`, SSE |
| Raw context window | 1,000,000 tokens |
| Effective input window | 950,000 tokens |
| Automatic compact threshold | 850,000 tokens (pinned in `models.json`) |
| Thinking mapping | none/minimal/low→disabled, medium→enabled, high/max→annotated 8,192, xhigh/ultra→annotated 16,384 |

## Supported capabilities

The runtime owns domain-neutral agent mechanics on the DeepSeek Chat Completions
path: multi-turn conversation and the agent loop; streamed text and reasoning;
tool discovery, invocation, and result continuation; dynamic tools, skills,
local plugins, and MCP; context accounting and automatic compaction; persisted
threads and resume; token and prompt-cache accounting; app-server stdio and
in-process host boundaries; and `codex exec` as a smoke/debug surface.

## Known limitations

- **Live scenarios are paid and opt-in.** The DeepSeek text/reasoning/tool live
  smoke and the 100K/300K/700K long-context checkpoints run only when explicitly
  invoked with a real `DEEPSEEK_API_KEY`; they are excluded from default CI. The
  offline suite covers the same logic deterministically against a mock.
- **`deepseek-v4-pro` is same-protocol but not separately gated.** It shares the
  DeepSeek data path and the 850K compact pin, but Runtime 0.0 maintains no
  separate release matrix for it.
- **Other providers and model slugs are best-effort.** Their code may remain
  compiled; failures specific to them do not block Runtime 0.0.
- **Non-goals remain out of scope** (§7.2): compile-time capability profiles,
  minimal dependency closure, enterprise multi-tenancy, additional providers,
  TUI completeness, realtime audio, V8/code-mode, remote plugin catalogs,
  ChatGPT/OpenAI product backends, Bazel release parity, SDK regeneration
  without a protocol change. Bazel and SDK CI jobs are the standing fork
  baseline and do not gate merges.
- **Compaction takes the local summarization path.** `supports_remote_compaction()`
  is `false` for every provider, so DeepSeek compaction summarizes via an
  ordinary chat/completions request — there is no dependency on a Responses
  `/compact` endpoint.
